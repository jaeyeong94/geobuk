# Geobuk Project Guidelines

## 작업 원칙: TDD (Test-Driven Development)

모든 기능 구현은 반드시 **테스트 우선(Test-First)** 방식으로 진행한다.
코드를 먼저 작성하고 테스트를 붙이는 방식은 금지한다.

### Red-Green-Refactor 사이클

1. **Red**: 실패하는 테스트를 먼저 작성한다
2. **Green**: 테스트를 통과하는 최소한의 코드를 작성한다
3. **Refactor**: 테스트가 통과하는 상태를 유지하며 코드를 개선한다

이 사이클을 반드시 준수하고, 각 단계를 건너뛰지 않는다.

---

## 테스트 작성 순서

기능을 구현할 때 아래 순서를 따른다:

### 1단계: 단위 테스트 (Unit Test)
- 개별 함수, 모듈, 클래스 단위로 테스트를 먼저 작성한다
- 외부 의존성은 mock/stub으로 격리한다
- 하나의 테스트는 하나의 동작만 검증한다

### 2단계: 네거티브 테스트 (Negative Test)
- 정상 경로(happy path) 테스트 후, 반드시 실패 경로를 테스트한다
- 다음 항목을 반드시 포함한다:
  - **잘못된 입력**: null, undefined, 빈 문자열, 타입 불일치, 범위 초과 값
  - **경계값**: 최솟값, 최댓값, 경계 직전/직후 값
  - **에러 상황**: 네트워크 실패, 타임아웃, 권한 없음, 리소스 부족
  - **동시성**: 경쟁 조건(race condition), 중복 요청, 순서 역전
- 에러 메시지와 에러 코드가 명확한지 검증한다

### 3단계: 퍼징 테스트 (Fuzz Test)
- 예측 불가능한 입력으로 시스템의 견고함을 검증한다
- 다음 전략을 적용한다:
  - **랜덤 입력 생성**: 무작위 문자열, 숫자, 바이너리 데이터
  - **변이 기반(Mutation-based)**: 유효한 입력을 변형하여 테스트
  - **속성 기반 테스트(Property-based testing)**: 입력에 관계없이 항상 성립해야 하는 불변 조건을 정의하고 검증
- 크래시, 메모리 누수, 무한 루프, 예외 미처리를 탐지한다
- 발견된 결함은 재현 가능한 단위 테스트로 변환하여 등록한다

### 4단계: 통합 테스트 (Integration Test)
- 모듈 간 연동, API 호출, DB 쿼리 등 실제 연결을 테스트한다
- 테스트 환경은 실제 환경과 최대한 동일하게 구성한다

### 5단계: E2E 테스트 (End-to-End Test)
- 사용자 시나리오 전체를 검증한다
- 브라우저 테스트는 **agent-browser 스킬을 CLI 방식으로** 사용한다:
  - Playwright MCP 도구(`mcp__plugin_playwright_playwright__*`)를 활용한다
  - 브라우저 시작, 페이지 이동, 요소 클릭, 폼 입력, 스크린샷 캡처를 자동화한다
  - 테스트 시나리오는 사용자 관점의 행위(로그인, 검색, 구매 등)로 작성한다
- CLI 기반 E2E 테스트는 실제 명령어 실행 후 출력과 종료 코드를 검증한다

## 계획 수립 규칙

기능 구현 전에 반드시 다음을 계획한다:

### 테스트 계획
- 구현할 기능의 **테스트 목록을 먼저** 작성한다
- 각 테스트의 목적, 입력, 기대 결과를 명시한다
- 네거티브 케이스와 엣지 케이스를 계획 단계에서 식별한다

### 구현 계획
- 테스트 목록을 기반으로 구현 순서를 정한다
- 가장 단순한 테스트부터 시작하여 점진적으로 복잡도를 높인다
- 한 번에 하나의 테스트만 통과시킨다

## 에이전트 작업 시 준수사항

### 코드 작성 전
1. 요구사항을 분석하고 테스트 케이스 목록을 작성한다
2. 네거티브 케이스, 경계값, 예외 상황을 반드시 포함한다
3. 테스트 파일을 먼저 생성하고, 실패하는 것을 확인한다

### 코드 작성 중
1. 테스트를 통과하는 최소 코드만 작성한다
2. 모든 테스트가 통과하면 리팩토링한다
3. 리팩토링 후 테스트가 여전히 통과하는지 확인한다

### 코드 작성 후
1. 테스트 커버리지를 확인한다
2. 누락된 네거티브/엣지 케이스가 없는지 검토한다
3. 퍼징 테스트로 예상치 못한 입력에 대한 견고함을 검증한다
4. E2E 테스트로 전체 시나리오가 동작하는지 확인한다

### 브라우저 E2E 테스트 실행
- `agent-browser` 스킬을 사용하여 CLI에서 브라우저 테스트를 수행한다
- Playwright MCP 도구를 통해 헤드리스 브라우저를 제어한다
- 테스트 결과는 스크린샷과 콘솔 로그로 증거를 남긴다

## 코드 재활용 원칙

새 코드를 작성하기 전에 반드시 기존 코드를 먼저 탐색한다.

### 기존 코드 우선 사용
- 코드 작성 전 `utils/`, `common/`, `lib/`, `helpers/`, `shared/` 등 범용 디렉토리를 탐색한다
- 동일하거나 유사한 기능의 함수가 이미 존재하면 **반드시 그것을 사용**한다
- 기존 함수가 요구사항에 80% 이상 부합하면 새로 만들지 말고 기존 함수를 확장한다

### 중복 코드 금지
- 동일한 로직을 2곳 이상에서 반복하지 않는다
- 중복이 발견되면 공통 함수로 추출하여 적절한 범용 디렉토리에 배치한다
- 복사-붙여넣기 후 일부만 수정하는 패턴은 금지한다 — 파라미터화하여 하나의 함수로 통합한다

### 범용 함수 관리
- 프로젝트 전반에서 사용되는 함수는 `utils/`, `common/`, `lib/` 등에 집중 관리한다
- 범용 함수는 특정 비즈니스 로직에 의존하지 않도록 순수하게 유지한다
- 범용 함수 추가/수정 시 기존 호출부에 영향이 없는지 확인한다

## 테스트 품질 기준

- 각 테스트는 독립적으로 실행 가능해야 한다
- 테스트 이름은 `[대상]_[상황]_[기대결과]` 형식으로 작성한다
- 하드코딩된 값 대신 의미 있는 테스트 데이터를 사용한다
- 테스트 실행 시간이 과도하게 길어지지 않도록 관리한다
- flaky 테스트는 즉시 수정하거나 격리한다

## 보안

### 입력 검증
- 사용자 입력, 외부 API 응답 등 시스템 경계에서 반드시 검증한다
- SQL 인젝션, XSS, 커맨드 인젝션을 방지하는 코드를 작성한다
- 쿼리 파라미터, 요청 본문, 헤더 값을 신뢰하지 않는다

### 시크릿 관리
- API 키, 비밀번호, 토큰 등을 코드에 하드코딩하지 않는다
- 환경변수 또는 시크릿 매니저를 사용한다
- `.env` 파일은 `.gitignore`에 반드시 포함한다

### 의존성 보안
- 외부 패키지 추가 시 알려진 취약점이 없는지 확인한다
- 사용하지 않는 의존성은 제거한다

## 에러 처리

### 에러 전파 원칙
- 에러를 삼키지(swallow) 않는다 — 적절히 전파하거나 로깅한다
- 빈 catch 블록을 금지한다
- 예상 가능한 에러(비즈니스 로직)와 예상 불가 에러(시스템 장애)를 구분하여 처리한다

### 에러 메시지
- 사용자향 메시지: 행동 가능한 안내를 제공한다 (예: "다시 시도해주세요")
- 개발자향 로그: 디버깅에 필요한 컨텍스트를 포함한다 (스택 트레이스, 입력값, 타임스탬프)
- 민감한 내부 정보(DB 쿼리, 서버 경로 등)를 사용자에게 노출하지 않는다

## Git 컨벤션

### 커밋 메시지
- Conventional Commits 형식을 따른다:
  - `feat:` 새로운 기능
  - `fix:` 버그 수정
  - `refactor:` 리팩토링 (기능 변경 없음)
  - `test:` 테스트 추가/수정
  - `docs:` 문서 변경
  - `chore:` 빌드, 설정 등 기타 변경
- 커밋 메시지는 한글로 작성하되, prefix는 영문을 유지한다

### 커밋 단위
- 하나의 커밋은 하나의 변경 목적만 담는다
- 기능 추가와 리팩토링을 하나의 커밋에 섞지 않는다
- 테스트와 구현 코드는 같은 커밋에 포함한다

---

## 코드 리뷰

### 에이전트 자동 리뷰
- 기능 구현 완료 후 보안 리뷰 에이전트를 실행한다
- 변경 파일이 10개 이상이거나 아키텍처 변경이 포함된 경우 코드 리뷰 에이전트를 추가 실행한다
- 리뷰에서 발견된 문제는 반드시 수정 후 완료 처리한다

### 리뷰 체크리스트
- 보안 취약점이 없는가
- 에러 처리가 적절한가
- 기존 유틸 함수를 활용했는가 (중복 코드 없는가)
- 테스트가 충분한가 (네거티브 케이스 포함)
